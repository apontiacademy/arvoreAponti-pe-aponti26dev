## What is being built

An internal tool for the company to manage "link tree" pages (Linktree/Bento.me/Beacons.ai style) — create, edit, and publish pages made of link/social/media blocks, with drag-and-drop reordering, per-page theming, and analytics. Access is restricted to authenticated employees only.

Explicitly **out of scope** — do not add SaaS-style features:
- Plans, pricing, subscriptions, billing
- Marketplace
- White-labeling
- Multi-tenant/multi-company support

## Architecture

- `src/main.tsx` → `src/App.tsx` → `AppProviders` (`src/app/providers.tsx`, `next-themes`' `ThemeProvider` — `attribute="class"`, `defaultTheme="system"`, `enableSystem`, `storageKey="apontilinkcenter-theme"` — wrapping TanStack Query's `QueryClientProvider`) → `RouterProvider` with the router from `src/routes/router.tsx`. `ThemeProvider` wraps everything (not just the authenticated app) specifically so the public `/:slug` route gets the same theme context as the admin routes.
- `src/components/theme/ThemeToggle.tsx` — a `DropdownMenu` (Claro/Escuro/Sistema) driven by `next-themes`' `useTheme()`, trigger icon swaps Sun/Moon based on `resolvedTheme`. Rendered in `Topbar` (admin) and `PublicPagePage` (fixed top-right corner, visitor-facing). `useTheme()` safely no-ops outside a `ThemeProvider` (returns `{ setTheme: () => {}, themes: [] }` per the library's own default context) — this is what already let `src/components/ui/sonner.tsx`'s `Toaster` call `useTheme()` before any `ThemeProvider` existed in the app, and is why component tests for `ThemeToggle`/`Topbar`/`PublicPagePage` don't need to wrap a real provider.
- `main.tsx` and `App.tsx` intentionally stay at `src/` root, not inside `src/app/` — only cross-cutting providers live in `src/app/`. Don't move the entrypoint without discussing it first; it was a deliberate call, not an oversight.
- Routing (`src/routes/router.tsx`): `/login` and `/:slug` (public page view, matched by `pages.slug` — **not** `profiles.username`; a page owner can have several pages, each with its own public URL) are public. Everything else is nested under `AuthGuard` (`src/features/auth/AuthGuard.tsx`, redirects to `/login` when there is no Supabase session) → `AppLayout` (`src/components/layout/AppLayout.tsx`: Sidebar + Topbar + Breadcrumb shell) → lazy-loaded page components (`React.lazy` + `Suspense`) under `src/routes/<screen>/`.
- Feature modules live in `src/features/<domain>/` — `auth` (`useSession.ts`, `AuthGuard.tsx`), `pages`: `usePages.ts` (list, scoped to owner, `updated_at desc`; takes an optional `options?: { allPages?: boolean }`, added 2026-07-20, which drops the `owner_id` filter for admin callers — `pages_admin_all` RLS decides what actually comes back, so this is purely a query-shape switch, not a trust boundary), `usePage.ts` (single page by id), `useCreatePage.ts`, `useUpdatePage.ts` (takes `{ id, values }` at mutate-time, not bound to the hook call, so one instance can update many rows — e.g. the publish toggle in a list), `useDeletePage.ts`, `useDuplicatePage.ts` (retry-on-`23505`-unique_violation loop appending `-copia`, `-copia-1`, ... to the slug — same pattern as the `handle_new_user` DB trigger — then copies the source page's `links` rows into the new page via a single bulk `.insert()`, **not** per-row inserts, since these are fresh rows rather than an update-only upsert like `useReorderLinks`), `pageSchema.ts` (shared Zod schema for create/edit forms), `slugify.ts` (title → URL slug, strips diacritics via `\p{Diacritic}` Unicode property escape); and `links`: `useLinks.ts` (list for a page, ordered by the `order` column), `useCreateLink.ts`, `useUpdateLink.ts` (same `{ id, values }`-at-mutate-time shape as `useUpdatePage`), `useDeleteLink.ts` (takes `{ id, pageId }` — `pageId` is only there to know which `['links', pageId]` query to invalidate, not sent to Supabase), `useReorderLinks.ts` (fires one `.update({ order })` per row via `Promise.all`, **not** a single `.upsert()` — `upsert`'s generated TS type requires all NOT NULL columns like `page_id`/`type` even though Postgres wouldn't need them for an update-only `ON CONFLICT`; per-row updates sidestep that), `linkTypes.ts` (the 14 `links.type` values with a lucide icon + label + optional value-field label/placeholder per type — the single source of truth the "add block" menu and each block's field labels read from). Follow this pattern for new domains (e.g. `analytics`) rather than dumping logic into route components.
- The link-block editor (`PageEditPage` + `src/routes/pages/components/LinkBlockCard.tsx`) does **not** use `react-hook-form`. Each `LinkBlockCard` owns its own `label`/`url` local state, autosaves on an 800ms debounce (same snapshot-ref pattern as `PageEditPage`'s own autosave — see below), and is keyed by `link.id` so React remounts a fresh card (fresh local state) if a different link ever occupied that slot, rather than needing prop-resync logic. Reordering: `Reorder.Group` (Framer Motion) wraps a list of `Reorder.Item`s (one per `LinkBlockCard`); `onReorder` only updates local visual state (`orderedIds`) — the actual `useReorderLinks` mutation fires from `onDragEnd` on the dragged item, not from every intermediate `onReorder` call, to avoid a mutation per pixel of drag movement. Each card uses `useDragControls()` + `dragListener={false}` so only its `GripVertical` handle (not the whole card, which contains text inputs) starts a drag.
- `src/features/links/linkValidation.ts` — plain functions (`validateLabelField`/`validateUrlField`), each backed by a Zod schema picked per `LinkType`, **not** a `react-hook-form` resolver like `pageSchema.ts` — the link-block editor isn't a form (see above), so validation is invoked directly from `LinkBlockCard`'s render rather than through RHF's validation pipeline. `LinkBlockCard` tracks `labelTouched`/`urlTouched` booleans (set `true` on blur) purely to gate when an error message is *displayed* (so a brand-new empty block doesn't show red text before the user has touched it) — the autosave effect itself checks the untouched-independent validity and simply skips scheduling a save while a field is invalid, so bad data is never silently persisted either way.
- `src/features/analytics/useAnalyticsSummary.ts` — takes an `ownerId`, fetches the owner's `pages` (`id`/`title`/`slug`/`is_published`) and all rows from the `analytics_summary` view in parallel, then merges them client-side keyed by `page_id`, defaulting `totalViews`/`totalClicks` to `0` for pages with no events yet. This left-join-in-JS is necessary because `analytics_summary` is a `GROUP BY` aggregate — it only has rows for pages with at least one recorded event, so a page with zero traffic would otherwise be silently missing from the dashboard rather than showing "0". No extra `owner_id` filter is applied to the `analytics_summary` query itself: its underlying `analytics` table RLS (`analytics_owner_select`) already scopes the aggregate to the caller's own pages, same trust as the RLS-then-`.eq()` belt-and-suspenders pattern used elsewhere. Takes an optional `options?: { allPages?: boolean }` (added 2026-07-20 for admin support, mirroring `usePages`'s shape below) that drops the owner filter on both the `pages` and `analytics_summary` queries — RLS's `analytics_admin_select`/`pages_admin_all` policies decide what actually comes back, same trust boundary as everywhere else. The mapped `PageAnalytics` type carries an `ownerId` field for this reason.
- `src/features/profiles/` — `useProfile.ts` (the logged-in user's own `profiles` row, including `role`; the one hook every screen that needs to know "is this user an admin?" reads from), `useUsers.ts` (lists all `profiles` — takes an `enabled` param, default `true`, so screens that only fetch it conditionally for admins, like `PagesListPage`/`AnalyticsPage`, can gate the request instead of firing it unconditionally for every user; RLS's `profiles_admin_select_all` policy is what actually restricts a non-admin caller to just their own row regardless of what the client asks for), `useSetUserRole.ts` (mutation wrapping the `set_user_role` RPC, `{ targetUserId, newRole }` → `{ target_user_id, new_role }`; invalidates both `['users']` and `['profile']` on success since a role change can affect the caller's own admin status; `onSuccess`/`onError` toasts live at the hook level, not per call site, so every consumer gets consistent feedback for free).
- `src/features/public/`: `usePublicPage.ts` (fetch a page by `slug`, filtered to `is_published = true` in the query itself — belt-and-suspenders alongside RLS, so an owner previewing their own unpublished page via its public URL still sees "not found", matching the disabled "Ver página pública" link elsewhere), `parseUserAgent.ts` (lightweight regex-based device/OS/browser sniffing — not a library, "good enough" for internal analytics, not exhaustive), `analytics.ts` (`recordAnalyticsEvent`, fire-and-forget from the UI, **never chain `.select()` after this insert** — see the RLS gotcha below), `resolveLinkHref.ts` (per-`link.type` href resolution: `wa.me/<digits>` for WhatsApp, `mailto:`/`tel:` for email/phone, profile-URL construction for Instagram/TikTok/Telegram when given a bare handle instead of a full URL, `null` for `title`/`text`/`pix` which aren't navigable). `PublicPagePage` reuses the existing `useLinks(pageId)` hook unchanged — RLS alone (not client code) is what narrows the result set to active links for anon vs. all links for the authenticated owner.
- **RLS + `RETURNING` gotcha (found while verifying analytics capture manually against the live project):** `analytics`'s insert policy (`analytics_public_insert`) lets anon insert when the target page is published, but its *select* policy (`analytics_owner_select`) only lets the page's owner read rows back. Postgres enforces the select policy on `RETURNING` too — so `supabase.from('analytics').insert(...).select()` (or anything that makes supabase-js request `Prefer: return=representation`) fails with a `42501 new row violates row-level security policy` for anonymous inserts, even though the insert itself is allowed. `recordAnalyticsEvent` deliberately does **not** chain `.select()` after `.insert()` — keep it that way, or add a `SECURITY DEFINER` RPC if the insert ever needs to return the created row.
- `src/components/ui/` — shadcn-generated primitives: `button`, `card`, `badge`, `skeleton`, `input`, `label`, `textarea`, `switch`, `dropdown-menu`, `alert-dialog`, `sonner` (Toaster wrapper — pulls in `next-themes` and `sonner` as transitive deps; no `ThemeProvider` is set up, so it always renders in `next-themes`' default "system" state, which is fine since there's no dark/light toggle yet), `sidebar` (+ its own transitive additions `sheet`, `tooltip`, `separator`, and the `useIsMobile` hook in `src/hooks/use-mobile.ts`), `breadcrumb`. No `form.tsx` — the `base-nova` registry doesn't ship one, so forms wire `react-hook-form`'s `register`/`handleSubmit` directly against `Input`/`Label`/`Textarea` instead of a Radix-style `Form` wrapper.
- `src/components/layout/` — `AppLayout` (wraps `Sidebar` + `SidebarInset` in a `SidebarProvider`, matching the shadcn Sidebar block's expected composition — the Sidebar/Topbar/Breadcrumb shell), `Sidebar` (rebuilt on the shadcn `Sidebar`/`SidebarHeader`/`SidebarContent`/`SidebarMenu`/`SidebarMenuButton` primitives — added 2026-07-17, replacing an earlier hand-rolled `<nav>` + plain `NavLink`s, per the "prefer 21st.dev/shadcn equivalents over hand-rolling" rule above; `SidebarMenuButton`'s `isActive` prop is set by comparing `useLocation().pathname` against each nav item's `to`, since NavLink's own `aria-current` only drives the link's own styling, not the shadcn button's `data-active` variant; renders via `render={<NavLink to={to} />}`, the same Base UI polymorphic pattern used elsewhere — the rendered element stays a real `<a>`, so tests still query `getByRole('link', ...)`), `Topbar` (now also renders `SidebarTrigger` — toggles the sidebar, and doubles as the mobile menu button since collapsed-on-mobile shows a `Sheet` drawer instead of the fixed rail), `Breadcrumb` (route-aware since 2026-07-17 — see below).
- **Route-aware Breadcrumb** (`src/components/layout/Breadcrumb.tsx`) — splits `useLocation().pathname` into segments and maps each to a label via a static `ROUTE_LABELS` dict (`dashboard`→"Dashboard", `pages`→"Árvores", `new`→"Nova árvore", `edit`→"Editar", etc.); the one dynamic case is `/pages/:id/edit`, where the `:id` segment's label is swapped for the page's real `title` via `usePage(params.id)` (falls back to "Carregando..." while that query is in flight) instead of showing a raw UUID. `useParams()` works here even though `Breadcrumb` renders from the `AppLayout` layout route, one level above the actual leaf route — React Router aggregates params across the whole matched route branch, not just the deepest match. Built on the shadcn `Breadcrumb`/`BreadcrumbItem`/`BreadcrumbLink`/`BreadcrumbPage`/`BreadcrumbSeparator` primitives; non-last crumbs are `BreadcrumbLink render={<Link .../>}`, the last is always a non-clickable `BreadcrumbPage`.
- `public/APONTI_SIMBOLO_RGB-01.svg` is the real Aponti brand mark (already tinted `#6518EA`, matching the primary token in `src/index.css`) — referenced by absolute path (`/APONTI_SIMBOLO_RGB-01.svg`) rather than imported from `src/assets/`, same pattern as the old Vite favicon it replaced. Used as: the browser favicon (`index.html`'s `<link rel="icon">`), the logo next to "ApontiLinkCenter" in `Sidebar`, and a standalone mark above the card on `LoginPage`. No `src/assets/` copy exists — don't duplicate it there.
- Destructive confirmations use `AlertDialog`. When the trigger lives inside a `DropdownMenu` (e.g. the "Excluir" row action in `PagesListPage`), do **not** nest `AlertDialogTrigger` inside a `DropdownMenuItem` — the menu closes on item click before the dialog can take over, causing focus/portal issues. Instead keep a piece of state (e.g. `pageToDelete`) set by the menu item's `onClick`, and drive a single controlled `AlertDialog` via `open`/`onOpenChange` outside the menu. When the trigger is a standalone button (e.g. the delete button on `PageEditPage`), `AlertDialogTrigger render={<Button variant="destructive" />}` works directly, no extra state needed.
- `src/lib/supabase.ts` — Supabase client, typed with `Database` from `src/lib/database.types.ts` (generated via the Supabase MCP `generate_typescript_types` tool — re-run and regenerate by hand after schema changes, there's no CI step for this yet), reads `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` from env and throws if missing. `src/lib/utils.ts` — shadcn's `cn()` helper.

- Tests live next to the code they cover (`*.test.tsx`/`*.test.ts`), plus `src/test/setup.ts` (jest-dom matchers) and a `src/test/smoke.test.ts` runner sanity check.

## Domain model (Supabase schema)

Live project: `arvore-aponti`, Supabase org "Aponti", region `sa-east-1`. Schema is versioned in `supabase/migrations/` (9 migrations so far: initial schema, an analytics RLS-bypass security fix, an RLS performance optimization, a `profiles` auto-provisioning trigger on `auth.users` insert, a follow-up revoking public EXECUTE on that trigger function, and 4 migrations for admin/user roles — see below) and was applied directly to the remote project via the Supabase MCP tools — **there is no local Supabase CLI setup** (no `supabase/config.toml`); see "Known gaps" below.
- `public.handle_new_user()` (SECURITY DEFINER, `search_path = public`) derives a `username` from the email local-part and de-duplicates on `unique_violation` (retry loop, not check-then-insert, to stay race-safe). EXECUTE is revoked from `public`/`anon`/`authenticated` — it's only meant to run via the `on_auth_user_created` trigger, not as a callable RPC. Follow this same lockdown pattern for any other SECURITY DEFINER function.
- `profiles.role` (`text not null default 'user' check (role in ('user','admin'))`, added 2026-07-20) plus `public.is_admin()` (SECURITY DEFINER, STABLE, `search_path = public`, checks `profiles.role = 'admin'` for `auth.uid()`) and the `set_user_role(target_user_id uuid, new_role text)` RPC (SECURITY DEFINER, checks `is_admin()` internally, rejects `new_role` outside `('user','admin')`, rejects `target_user_id = auth.uid()` to block self-demotion even via direct RPC call, granted to `authenticated` only). `profiles.role`'s `UPDATE` grant is revoked from `authenticated` and re-granted column-scoped on everything except `role` — Postgres column-level `REVOKE` does **not** narrow a pre-existing table-level `GRANT`, so this required an explicit table-level revoke + column-level re-grant, not just a bare `REVOKE UPDATE (role)`. `set_user_role` is the only path that can change a role; a client writing `role` directly through the normal `profiles` update path fails, by design.

Tables: `profiles` (1:1 with `auth.users`), `pages` (owned by a profile via `owner_id`), `links`, `social_links`, `analytics`, `themes` (5 seeded presets: Minimal, Dark, Glass, Corporate, Modern). Each page ("árvore") belongs to a profile and has many links/social links, one theme, and analytics events.

- RLS is enabled on every table. Pattern: owner has full access (`auth.uid() = owner_id`, wrapped in a `(select auth.uid())` subselect per-policy for query-plan performance — do not regress this when adding policies), public/anon can only `select` from **published** pages, their active links, and all their social links. `themes` is public-read. The `analytics_summary` view uses `security_invoker = true` so it respects the caller's RLS rather than the view owner's. Admin access rides alongside these as separate, additive policies (`pages_admin_all`/`links_admin_all`/`social_links_admin_all` are `FOR ALL using (is_admin())`, `analytics_admin_select`/`profiles_admin_select_all` are `FOR SELECT using (is_admin())`) rather than folding admin logic into the existing owner policies — keeps the owner-scoped policies unchanged and the admin bypass auditable as its own set of rules.
- Indexes exist on all foreign keys used in RLS lookups (`pages.owner_id`, `links.page_id`, `social_links.page_id`, `analytics.page_id`/`link_id`, `pages.theme_id`).
- Analytics captures `event_type` (`view`|`click`), device, OS, browser, referrer, timestamp, per page and optionally per link; `analytics_summary` aggregates `total_views`/`total_clicks` per page. Public insert into `analytics` is restricted to published pages only (fixed in the second migration — the original policy allowed inserting analytics rows for unpublished/private pages, an RLS bypass).
- Not yet built: RPC functions, triggers, most-viewed/most-clicked-link aggregates beyond the basic view.

## Screens / feature surface

Login, Dashboard, Page list (search + duplicate), Create/Edit page (drag-and-drop editor), Public page view, Analytics, Profile, Settings, 404. Route placeholders for all of these exist under `src/routes/`. `Login`, `Dashboard`, Page CRUD, the link-block editor, the public page view, Analytics, and Settings (admin section only, see below) now have real behavior; Profile still renders static placeholder text pending its build-order slot.

- **Login** (`src/routes/login/LoginPage.tsx`) — email/password form (`react-hook-form` + `zod` via `zodResolver`), calls `supabase.auth.signInWithPassword`, shows field-level validation errors and a generic "Email ou senha inválidos" alert on auth failure, navigates to `/dashboard` on success. Password field has a show/hide toggle (`Eye`/`EyeOff` from lucide, `aria-label` swaps between "Mostrar senha"/"Ocultar senha"). There is no self-serve signup screen — employees are provisioned some other way (see "Known gaps").
- **Dashboard** (`src/routes/dashboard/DashboardPage.tsx`) — reads `useSession()` + `usePages(session?.user.id)`. Three stat cards (total/publicadas/rascunhos), an "Árvores recentes" card (up to 5, most-recently-updated first) with a published/rascunho `Badge`, a skeleton loading state, an empty state with a "Criar primeira árvore" CTA, and an error message if the query fails. "Nova árvore" and both empty-state CTAs link to `/pages/new`.
- **Page list** (`src/routes/pages/PagesListPage.tsx`) — `usePages`, client-side search filter (title or slug, case-insensitive — no server round-trip), skeleton/empty/error states like the Dashboard. Each row has a `DropdownMenu` with Editar, "Ver página pública" (disabled via `disabled={!page.is_published}` until published, opens `/${slug}` in a new tab), Duplicar, Publicar/Despublicar, and Excluir (destructive, routes through the confirmation `AlertDialog` described in "Architecture" above). All mutations show a `toast.success`/`toast.error`. When the logged-in user is an admin (`useProfile().data?.role === 'admin'`), `usePages(session?.user.id, { allPages: true })` returns every user's pages (RLS decides what actually comes back), and each row shows a "por {username}" owner annotation resolved client-side against `useUsers(isAdmin)` — not a SQL join, since expected page/user counts don't warrant one yet.
- **Create page** (`src/routes/pages/PageNewPage.tsx`) — title/slug/description form. Slug auto-derives from title via `slugify()` until the user edits the slug field directly (tracked with a `slugEdited` boolean state, not a ref, since it needs to trigger the effect that stops the sync). On submit, `useCreatePage` looks up the `minimal` theme's id and sets it as `theme_id` (theme selection UI is deferred to the personalization/editor phase). A `23505` (unique_violation) error on `slug` is mapped to a friendly "Essa URL já está em uso." field error via `setError`; other errors fall back to a toast.
- **Edit page** (`src/routes/pages/PageEditPage.tsx`) — fetches via `usePage(id)`. Title/slug/description **autosave**: a debounced (800ms) effect compares the current form values against a "last saved" snapshot (a ref, updated on load and after each successful save) and calls `useUpdatePage` only when they actually differ — this avoids firing a save on initial mount/data-load, only on genuine edits. A small status line reflects `idle`/`saving`/`saved`/`error`. The publish `Switch` and the delete `AlertDialog` are separate, immediate actions (not part of the debounce). Same `23505` → friendly slug error handling as create. Below the page-details card is the **link-block editor** (see next bullet).
- **Link-block editor** — lives inside `PageEditPage`, in a "Blocos" card. An "Adicionar" `DropdownMenu` lists all 14 block types (from `linkTypes.ts`, each with an icon); picking one calls `useCreateLink` with the next `order` index. Each block renders as a draggable `LinkBlockCard`: a type icon, a label input (a `Textarea` for `text`, an `Input` for everything else — `title`/`text` have no URL field, every other type does, with a per-type label/placeholder, e.g. "Número (com DDI)" for WhatsApp, "Chave Pix" for Pix), a per-card autosave status, an `is_active` `Switch` (soft-hide without deleting), and a delete `AlertDialog`. Drag handle is a dedicated `GripVertical` icon (not the whole card), so clicking into the text inputs never starts a drag.

- **Public page** (`src/routes/public/PublicPagePage.tsx`, route `/:slug`) — `usePublicPage(slug)` + `useLinks(page?.id)`. Renders `page.title`/`page.description` and the active blocks in order via `PublicLinkBlock` (`src/routes/public/components/PublicLinkBlock.tsx`): `title`/`text` render as plain heading/paragraph (not clickable), `image`/`video` render inline (`<img>`/`<video controls>`), `pix` renders as a button that copies the key to the clipboard (`toast.success`) instead of navigating, everything else renders as an `<a target="_blank">` with its href resolved by `resolveLinkHref`. Records one `view` event on mount (guarded by a ref keyed on `page.id`, so it doesn't double-fire — relevant in dev under `StrictMode`) and one `click` event per block interaction, via `recordAnalyticsEvent`. Loading skeleton and a generic "não existe ou não está mais disponível" message double as the 404 state for both truly-missing slugs and unpublished ones (indistinguishable to a visitor, by design).

- **Analytics** (`src/routes/analytics/AnalyticsPage.tsx`) — reads `useSession()` + `useAnalyticsSummary(session?.user.id)`. Two stat cards (total views/clicks summed across all the owner's pages) and a "Desempenho por árvore" card listing every owned page sorted by most-viewed-first, each row showing view count, click count, a published/rascunho `Badge`, and (only when published) an external-link icon to open the public page in a new tab. Same skeleton/empty/error states as Dashboard and Page list. Unlike those, this list isn't paginated or searchable — acceptable for now given expected page counts per employee, revisit if that changes. Same admin-mode pattern as Page list: `useAnalyticsSummary(session?.user.id, { allPages: isAdmin })` plus a client-side owner annotation from `useUsers(isAdmin)` (keyed by `page.ownerId`, the camelCase field this hook's mapped type uses — not the same casing as `PagesListPage`'s raw `owner_id`, intentionally, since one comes from a mapped hook result and the other from the raw table row).
- **Settings** (`src/routes/settings/SettingsPage.tsx`) — reads `useProfile(session?.user.id)`; for a non-admin this still renders only the page heading (Profile-style settings themselves aren't built yet). For an admin, a "Usuários" `Card` (`UsersSection`) lists every profile via `useUsers(true)`, each row showing a role `Badge` and a `Switch` to promote/demote. Promoting a basic user is immediate; demoting an admin routes through an `AlertDialog` confirmation (same controlled-state pattern as page deletion in "Architecture", since the trigger here is a `Switch`, not a standalone button or menu item) before calling `useSetUserRole()`. The admin's own row has its `Switch` disabled (`aria-disabled`, not the native `disabled` attribute — the shared `Switch` primitive is a Base UI `<span>`, not a native form element) so nobody can self-demote from the UI, backing up the same guard already enforced by the `set_user_role` RPC. `useSetUserRole` shows a `toast.success`/`toast.error` on every role change, same convention as other mutations in this app.

Duplicating a page copies the `pages` row itself (title, slug + `-copia` suffix, description, theme, settings) with `is_published` forced to `false`, plus every row in `links` for that page (same label/url/payload/type/order/is_active, re-pointed at the new `page_id`) — `useDuplicatePage` fetches the source links ordered by `order` right after the new page insert succeeds, then does one bulk `.insert()` of the copies. `social_links` is still not duplicated (that table has no UI/hooks yet at all — see "Known gaps").

Editor block types (matches the `links.type` check constraint, all implemented in both the editor and the public rendering): Title, Text, Link, WhatsApp, Instagram, TikTok, Telegram, YouTube, Spotify, Pix, Email, Phone, Image, Video. Image/Video only store a URL for now — there's no upload flow (Supabase Storage isn't wired up); rendering is just a plain `<img>`/`<video>` tag, no lazy-loading/error-fallback handling yet.

Per-page personalization: primary/secondary color, font, background (color/image/gradient), radius, shadow, width, spacing, theme preset (Minimal, Dark, Glass, Corporate, Modern), light/dark mode.

## Working conventions

- Always produce production-ready code — no placeholder/temporary implementations (route placeholders from Phase 1 are the sanctioned exception, tracked above — replace them as their build-order slot comes up, don't add new placeholders elsewhere).
- Favor small, reusable, strongly-typed components and hooks; avoid duplication.
- Apply SOLID where it genuinely fits a component/hook boundary — don't force it onto simple CRUD code.
- Editor UX must support autosave while editing and a confirmation step before destructive actions (deleting links/pages).
- Performance: lazy-load routes, code-split the editor, memoize expensive renders, keep TanStack Query keys/queries scoped to avoid over-fetching.
- New Supabase schema changes go in a new file under `supabase/migrations/`, applied via the Supabase MCP tools, and must ship with RLS policies (not bare tables) — follow the `(select auth.uid())` subselect pattern used in the existing policies.
- Gitflow: `main`/`develop`/`staging` are permanent branches; work happens on `feature/*` branches off `develop`, merged via PR (CI runs lint/typecheck/test/build on the PR).
